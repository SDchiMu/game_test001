# CLAUDE.md — 项目指令

> 克隆新项目后复制本文件到根目录，按实际技术栈修改即可。

---

## 一、通用行为规范

### 回答问题
- **先说结论**（一句话总结），再展开细节
- 涉及代码时提供完整的可运行示例
- 注明适用的语言/框架版本

### 代码审查
- 输出顺序：**明确错误 → 边界情况 → 性能/可读性建议**
- 每条建议附带修改前后的代码对比
- 安全隐患优先高亮
- 问题分级：🔴 严重 / 🟡 中等 / 🔵 建议

### 编写测试
- 覆盖面：正常路径 + 至少两个边界路径 + 一个异常路径
- 外部依赖使用 mock，不产生真实副作用

### 处理错误报告
- 先尝试复现问题
- 根据实际代码分析原因，不凭空猜测
- 给出修正后代码并解释修改理由

### 代码有语法错误时
- 自动修复并展示修正后的代码
- 用注释说明修改原因

## 二、需求理解与沟通

以下规则帮助我在动手前准确理解你的意图，减少返工。

### 当需求模糊时
- **主动提问**：如果我需要更多信息才能开始，我会先问清楚，不会假设
- 问题应具体、可回答，避免开放式的"你想怎么做"

### 当任务复杂时
- **先出计划**：改动涉及多个文件或模块时，我先列计划等你确认再动手
- 计划包括：涉及文件、改动思路、潜在风险

### 当有多个可行方案时
- 我会列出各方案的**权衡点**（复杂度 / 性能 / 可维护性 / 学习成本）
- 给出我的**推荐**和理由，但不是替你做决定

### 当我的建议与已有代码矛盾时
- 我会指出现有代码的做法和我的建议之间的差异
- 问你倾向于保持一致还是重构
- **代码库已有的模式优先于我个人偏好**

### 在结束时
- 总结改了**什么**、**为什么改**、**影响范围**
- 如果有未完成的部分或后续待办，明确列出

## 三、通用代码规范

### 函数与方法
- **单一职责**：一个函数只做一件事。超过 30-40 行应考虑拆分
- 命名反映行为：`process_data` 太模糊，`validate_email` 才清楚
- 返回值类型一致，不出现有时返回 `str` 有时返回 `None`

### 命名
- 项目已有的命名风格优先，不混用
- 文件名反映主要职责
- 布尔变量/函数用肯定语气：`isActive`、`hasPermission`、`shouldRetry`

### 注释
- 注释说 **为什么**（why），代码说 **做什么**（what）
- 不要给显而易见的东西写注释（`i += 1  // 给 i 加 1`）
- TODO 必须附带上下文：`// TODO: 需要处理分页，目前只取了前 100 条`
- 不留注释掉的代码，用版本控制历史找回

### 错误处理
- **不要吞异常**：空的 `try/catch` 或 `except Exception: pass` 必须有注释说明为什么忽略
- 宁可尽早失败（fail fast）也不要让错误状态蔓延
- 对外接口做参数校验，对内断言

### 状态与配置
- 硬编码的数字 / 字符串提取为具名常量
- 环境相关配置（URL、密钥、路径）放在环境变量或配置文件中

### 安全底线
- 不要硬编码密钥、令牌、密码
- 用户输入必须校验和转义（SQL 注入、XSS、命令注入）
- 文件路径操作防止路径穿越

## 四、工程实践

### 版本控制
- 提交信息格式：`<类型>: <简短描述>`（类型：`feat` / `fix` / `refactor` / `docs` / `test` / `chore`）
- 一个提交只做一件事
- 不提交：编译产物、依赖包、 IDE 配置文件（已在 `.gitignore` 中的）

### 测试哲学
- 核心逻辑必须有测试
- 测试是**行为的契约**，不是实现细节的快照
- 测试应该不依赖网络、数据库等外部资源（或通过 mock 隔离）

### 依赖管理
- 引入新依赖前三思：是否真的需要？能否用标准库解决？
- 明确记录依赖及用途
